System Architecture Approaches

Three ways to structure the ESP32 thermal management firmware

A

FreeRTOS Task-Based (Recommended)

Separate RTOS tasks for TempSensor, FanControl, WebServer, ButtonHandler. Queue-based inter-task communication. Timer for sampling rate.

Pros

  • Clean separation of concerns
  • No delay() — pure non-blocking
  • Easy to debug per-task
  • Web server stays responsive
  • Scales well for capstone

Cons

  • More code boilerplate
  • Shared data needs mutex
B

Super-Loop + Timer Callbacks

Main loop() handles web server polling. ESP timer callbacks for temp sampling and fan control. Simple state machine for modes.

Pros

  • Simpler code structure
  • Lower memory overhead
  • Good for smaller projects

Cons

  • Web server can block
  • Timer conflicts at high freq
  • Harder to extend later
C

Event-Driven with State Machine

Central event queue. Components emit events (temp_reading, button_press, web_command). State machine governs transitions between idle/cooling/manual/error.

Pros

  • Very predictable behavior
  • Easy to add new events
  • Great for safety-critical logic

Cons

  • Most complex to implement
  • Event queue can overflow
  • Overkill for this scope

Click an option above to select. I recommend Approach A (RTOS Tasks) — it's the cleanest for a capstone: each subsystem is independently testable, the web server stays responsive via its own task, and the structure maps directly to your report's architecture section.
